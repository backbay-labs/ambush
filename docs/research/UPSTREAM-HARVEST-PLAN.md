# Ambush Upstream Harvest Plan

## 1. The big picture

The single most important realization is this: **Ambush's "Chio" is not a thing it has to build â€” it is a thing it has to reunite with.** The 157-line `src/main/governance/chio-governor.ts` shim shells out to `chio mcp serve/wrap` and `chio receipt list --json` against a binary that does not exist in the repo and, worse than the brief states, **fails open while its own header comment claims fail-closed** (`configure()` on a missing binary returns `detail: 'Agents run ungoverned'`; `wrapMcp()` at L83-84 returns the inner command unchanged; `DEFAULT_POLICY` allows `exec`/`write`/`edit`/`move` and denies only `delete`). That binary *exists*: it is `crates/products/chio-cli` in the ARC/Chio repo, with a genuinely fail-closed `code_agent.yaml` and an MCP gating loop. The governance layer Ambush markets as non-repudiation has a real, signed, proptest-covered upstream.

Three facts compound into the strategy:

- **The proof-room IS the attestation moat.** Chio's `crates/products/chio-proof-room` already implements, end to end, what the brief calls "Export Attestation" + standalone "ambush verify": a `bundle.schema.json` manifest that hash-binds every artifact, a detached DSSE envelope (`bundle-signature.dsse.json`), an offline verifier that recomputes all hashes and re-checks signatures against pinned trust roots, a receipt-coverage matrix, and shipped negative-case fixtures that prove the verifier rejects tampering. This is the in-toto/Sigstore-aligned receipt chain, built and tested.
- **ClawdStrike is the engine's parent.** `engine/crates/swarm-crypto` is already a fork of `hush-core` (canonical/merkle/hashing/signing + a `Signer` trait), and `swarm-guard` copied ClawdStrike's `GuardResult/GuardAction/Severity/Guard` types verbatim. The seams already line up â€” harvesting is reunification, not porting.
- **Both upstreams are Apache-2.0**, identical to the engine's license. Every Rustâ†’engine lift needs only NOTICE attribution. The MIT control plane can subprocess Apache-2.0 binaries cleanly.

What reuniting unlocks: the fail-open shim becomes a real fail-closed governor; `swarm-crypto`'s forked-but-inert crypto gains the one missing `SignedReceipt` primitive that turns it into an attestation layer; the demo's "show an ALLOW and a DENY receipt, both signed and offline-verifiable" painkiller becomes real instead of vapor; and the NDA/burst client deliverable (a redacted-but-still-verifiable attestation archive) is a near-copy of `export.rs`.

## 3. The Chio reunification

This is the highest-value harvest, and it has a precise shape. Today `chio-governor.ts` invokes `chio mcp serve --policy <p> -- <inner>` and `chio receipt list --admin-all --json`. The matching real binary is `arc/crates/products/chio-cli/src/bin/chio.rs`, whose `src/cli/mcp/wrap.rs` (`run_wrap_with_gate`, ~250 lines) gates every `tools/call` through a default-deny `VerdictGate` emitting `urn:chio:error:*` reason codes, and whose `code_agent.yaml` is strictly better than `DEFAULT_POLICY` (forbidden_path `.env`/`.ssh`/`.aws`/`*.pem`/`id_rsa`; shell forbidden regex `rm -rf /`, `curl|sh`, `git push --force`, `sudo`, fork bombs; secret_patterns; patch_integrity; capability allowlist with TTLs).

**Do NOT ship the whole `chio-cli`.** Its `default-run = chio` pulls ~50 workspace crates including web3, federation, and market â€” verified via `chio-proof-room/Cargo.toml` pulling 17 `chio-*` crates. Two viable reunification paths:

- **Path A (recommended for v1): back the shim with the engine's own surface.** Adopt `code_agent.yaml` as `engine/rulesets/code-agent.yaml` immediately (trivial data port), and re-point `chio-governor.ts` from the phantom `chio` binary to the engine's `swarmctl serve`/guard-eval, which already speaks the 5 sync guards. The `wrap.rs` per-call ALLOW/DENY loop is the reference for the streaming `evt:receipt` + DenyToast.
- **Path B (when a real daemon exists): subprocess a trimmed `chio`** exposing only `mcp wrap/serve` + `receipt list`, run as an Apache-2.0 sidecar the MIT control plane shells. Cleaner non-repudiation, heavier build.

**Policy format:** adopt ClawdStrike `HushSpec` (`crates/libs/hushspec/src/{validate,merge,resolve,signing}.rs`) incrementally as the authoring layer â€” start with `validate.rs` (RE2-safe, ReDoS-proof regex) and the schema, then `merge/resolve` for `extends` inheritance (an NDA child policy that never weakens a parent), then signing. `code_agent.yaml` is the content; HushSpec is the validated, signable substrate underneath it.

**Wiring proof-room into Export Attestation + ambush verify** (the core move): create a new `engine/crates/swarm-attest` and carve out â€” not import â€” the leaf logic from `chio-proof-room/src/bundle_a.rs`. The DSSE verify (`verify_bundle_signature` L830-892, `dsse_pre_auth_encoding` L931-943 â€” 12 self-contained lines, `trusted_bundle_signer_keys` L894-929 with the dual fail-closed gate: keys must appear in both `trust-roots.json` AND an env-pinned trusted-signer set) ports mechanically because it depends only on `PublicKey`/`Signature`/`sha256_hex`, all interface-compatible with `swarm-crypto`. Bring `receipt_coverage.rs` (279 LOC, allow/deny/failure matrix with kernel-key sig verification, `require_full_matrix` forbidding exclusion of allow/deny) and `bundle.schema.json` (rename id to `ambush.attestation.bundle.v1`, keep the artifactRef/claim/receiptCoverage/signature `$defs`). **Stub the orchestrator's `verify_source_verifier_report` routing** â€” that is the trap that drags in the 14-crate domain stack. For `ambush verify` itself, fuse ClawdStrike's `cmd_verify` (stable JSON output, exit codes, fail-closed parse) with Chio's 6-bucket exit-code taxonomy (10 required-claim, 20 integrity, 30 parse/schema, 40 negative-case-passed, 50 unsupported, 60 release-truth) into a `swarm-cli verify <bundle>` subcommand. Ship the negative-case self-test (`bundle_a.rs` L169-243): mutate the bundle, re-verify, assert it fails with the expected code â€” turning "tamper-evident" from a claim into a checked property inside every attestation.

**The one schema decision that must be made once:** ClawdStrike `receipt.rs` is the internal model; the Chio DSSE bundle is the export envelope; `chio-ts` is re-pointed to verify *that chosen schema*. Adopting `hush-core` `SignedReceipt` *and* `ChioReceipt` *and* the verification-bundle without picking one canonical field set guarantees canonical-JSON byte drift and cross-language verification failure. Lock it with shared corpora: `hush-core/tests/.../jcs_vectors.json` + `version_cases.json` + Chio `chio-conformance` vectors, run in both the Rust crate and the TS verifier.

**TS-side license nuance:** `chio-ts` (`sdks/typescript/chio-ts/src/invariants/{receipt,crypto,json,manifest}.ts`, pure `node:crypto`, Ed25519/P256/P384, content-addressed receipt IDs) fixes the confirmed "trusts `chio receipt list` with no client-side verification" gap. **npm-depend on `@chio-protocol/sdk`** rather than vendoring â€” vendoring Apache-2.0 source into the MIT npm package makes it mixed-license. If you must vendor, isolate in a labeled Apache-2.0 directory with NOTICE; reuse `json.ts`+`crypto.ts` verbatim and rewrite `receipt.ts`'s field list to the chosen schema.

## 4. Swarm-governance tie-in

Chio's `crates/kernel/chio-swarm-authority` is the real spine for "non-repudiation for everything the swarm touches." Verified clean crate (deps: `chio-core-types`, `serde`, `thiserror`), it defines a recursive task-graph DAG where each child task is authorized by a signed `SwarmContinuationToken` cross-bound to a delegation witness chain, a route-plan receipt, a budget-pool allocation, a revocation epoch, and a terminal graph receipt. Trust keys are pinned via `did:chio:<hex>` issuers and `CHIO_SWARM_TRUSTED_WITNESS_KEYS` â€” never request-supplied, the exact opposite of the shim.

The mapping to Ambush is direct: **one signed continuation token per Vector hop.** Each Vector becomes a task-graph node; its tool launch carries a `chioSwarm` context referencing that Vector's continuation token, bound to its scope, a budget lease, and the live revocation epoch. The `admission_hook/swarm_authority.rs` (265 LOC) pattern is the blueprint for the reborn fail-closed daemon: evidence referenced by id+sha256, re-hashed on load, full bundle verified against pinned keys, then SingleUse continuations consumed (replay-denied). It **explicitly rejects request-smuggled `trustRoot`/`witnessKeys`** â€” wire this anti-smuggle denylist verbatim. Recall gets cryptographic teeth: instead of only SIGKILLing the PTY, Ambush's recall UI issues a signed `SwarmRevocationEpoch` (pattern in `arc/crates/trust/chio-revocation-oracle/tests/swarm_revocation_e2e.rs`) that the trust-kernel enforces on the next admission.

**Pragmatic scope for v1:** Ambush Vectors share a fixed per-operation scope, so collapse the multi-hop attenuation witness chain to single-hop trivial proofs (the verifier already feature-gates `multi_hop_witness_chains`), avoiding the heaviest transitive dependency â€” `attenuation.rs` (~1,099 lines). Keep the continuation-token/route/budget/revocation spine; defer real attenuation until nested delegation is a product need. Re-point the verifier's crypto at `swarm-crypto` via a small `sign_canonical/verify_canonical/sha256` trait rather than vendoring `chio-core-types` (the two-crypto-stack trap). Copy the v1 schemas (`spec/schemas/chio-swarm/v1/*`) and the `r_t03` golden + 6-mutation negative fixtures (`fixtures/proof-room/swarm-authority/valid-recursive-delegation/*`) as free acceptance tests; regenerate signatures against an Ambush issuer key.

## 5. Capability brainstorm

Grounded new capabilities the upstreams suggest:

- **SIEM/audit export tier.** `arc/crates/observability/chio-siem` fans signed receipts to Splunk/Elastic/Datadog/OCSF/CEF with DLQ, rate limiting, and PagerDuty alerting, paired with `spec/audit-log/export-schema.v1.json`. A ready Org/Audit-tier "receipts into your SIEM" feature â€” adapt the exporter trait + OCSF mapping, swap the receipt source to swarm-spine.
- **Receipt provenance DAG.** `chio-lineage` (`schemas/lineage-graph.v1.json`) turns the AuditStream into a navigable chain-of-custody graph â€” what was authorized, by which capability, in what order â€” strengthening the differentiation vs competitors' app-level logging and complementing Ambush's IntelGraph rail.
- **Per-lane budget governance.** `arc/crates/economy/chio-metering`'s budget hierarchy + enforcement directly addresses the brief's cost doom-loop; re-point the data model from tool-access cost to LLM token spend.
- **Log redaction defense-in-depth.** `chio-log-redact`'s `redacted!()` tracing layer prevents secrets/PII from leaking into the live PTY AuditStream â€” cheap complement to EnvScrub.
- **Threat-intel enrichment for review.** ClawdStrike `async_guards/threat_intel/{snyk,virustotal,safe_browsing,spider_sense}` lets a Vector reviewing a PR enrich findings with CVE/dependency/URL intel; the `circuit_breaker/cache/rate_limit` runtime is reusable regardless of providers.
- **Sandbox attestation schema.** ClawdStrike `sandbox/attestation.rs` emits a `SandboxAttestation` into the receipt chain â€” adopt the schema even if you swap `nono` enforcement for a container runtime.

**Ignore:** the agent economy/marketplace/web3 settlement, EAS/Ethereum anchoring (contradicts local-first, adds liability), protocol bridge crates (MCP/A2A/ACP â€” Ambush governs CLI agents, not protocol dispatch), and `chio-guard-sdk` WASM guards (competes with the engine's native guards).
